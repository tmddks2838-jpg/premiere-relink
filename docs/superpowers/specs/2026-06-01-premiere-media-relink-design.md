## 1. 배경 & 문제 정의

프리미어 프로(Premiere Pro)는 프로젝트(`.prproj`)가 참조하는 미디어 파일의 **절대 경로**를 저장한다.
따라서 파일 위치를 옮기거나 폴더를 정리하면 미디어가 "오프라인(Media Offline)" 상태가 되고,
사용자가 매번 수동으로 재연결(Link Media)해야 한다.

특히 **오래된 프로젝트**는 다음 이유로 재연결이 어렵다.

- 정리하면서 파일·폴더 위치가 바뀜
- 일부 이미지/영상이 실제로 삭제됨
- 같은 이름의 파일이 여러 위치에 흩어짐

### 실측 검증 (사용자 실제 프로젝트)

`0430 김보미 오부부` 프로젝트(`~/Documents/자동저장/...`)를 분석한 결과:

- `.prproj`는 **gzip으로 압축된 XML**이며, 미디어 경로는 `ActualMediaFilePath`(7,626개), `FilePath`(15,403개) 태그에 **평문 절대 경로**로 저장됨.
- 동일 미디어가 **3가지 경로 형태**로 흩어져 있음:
  - `/Users/gimseung-an/Desktop/무제 폴더/0430 김보미 오부부/...` (원래 위치)
  - `/Volumes/Macintosh HD/Users/gimseung-an/Desktop/무제 폴더/...` (볼륨 접두 변형)
  - `/Users/gimseung-an/Desktop/0430 김보미 오부부/...` (정리하며 옮긴 위치)
- → **폴더 통째 이동**(무제 폴더 밖으로)이 경로를 깨뜨린 정황이 데이터에 그대로 남아 있음. 페인포인트가 실재함을 확인.

## 2. 목표 & 비목표

### 목표 (v1)

- 클로드 코드에게 **"이 프로젝트 미디어 재연결해줘"** 라고 말하면, `.prproj`를 분석해 깨진 경로를 찾아 자동/반자동으로 재연결한다.
- **확실한 매칭은 자동**, **애매한 매칭은 사용자에게 확인**, **못 찾은 파일은 건드리지 않고 리포트**한다.
- 원본 프로젝트를 **절대 손상시키지 않는다** (백업 + 새 파일 저장 + 수술적 치환).

### 비목표 (v1에서 안 함 — YAGNI)

- 프리미어를 켜서 공식 API로 연결 (B안) — 채택 안 함.
- GUI / 배포용 패키징 — v1은 클로드 스킬. 단, **엔진을 분리 설계**하여 추후 확장 기반만 마련.
- 삭제된 미디어의 복구·재생성.
- 프리미어 외 다른 NLE(DaVinci, Final Cut 등) 지원.

### 미래 확장 (v2+, 이번 범위 아님)

- v2: 동일 엔진 위에 **터미널 CLI** 껍데기 (`relink 프로젝트.prproj`).
- v3: 동일 엔진 위에 **드래그&드롭 맥 앱/GUI**.
- → 핵심은 **엔진(순수 로직)을 껍데기(인터페이스)와 분리**해, 인터페이스만 교체하며 재사용하는 것.

## 3. 전체 아키텍처 — "엔진 + 껍데기" 분리

```
┌─ 껍데기 (v1: 클로드 스킬 / v2: CLI / v3: GUI) ─┐
│  • 실행 명령 수신                              │
│  • 미리보기 표시, 애매한 매칭 사용자 확인        │
│  • 결과 리포트                                 │
└──────────────┬────────────────────────────────┘
               ↓ 호출 (잘 정의된 함수 인터페이스)
┌─ 엔진 (순수 파이썬, 클로드/UI 비의존, 재사용 핵심) ─┐
│  ① reader   : .prproj 해제 + 미디어 경로 추출      │
│  ② detector : 오프라인(누락) 경로 탐지             │
│  ③ indexer  : 검색 폴더 색인 (파일명→실제 위치)     │
│  ④ matcher  : 깨진 미디어 ↔ 후보 매칭 + 신뢰도 산정 │
│  ⑤ writer   : 경로 치환 후 저장 (백업 우선)        │
└────────────────────────────────────────────────┘
```

**설계 원칙**: 엔진의 각 모듈은 단일 책임을 갖고, 입출력이 명확한 함수로 통신한다.
엔진은 클로드/터미널/GUI 어떤 껍데기에서도 동일하게 호출 가능해야 한다 (인터페이스 비의존).

### 프로젝트 디렉토리 구조

```
premiere-relink/
├── docs/superpowers/specs/   # 설계 문서
├── engine/                   # 순수 파이썬 엔진 (재사용 핵심)
│   ├── reader.py             # ① .prproj 읽기/해제, 경로 추출
│   ├── detector.py           # ② 오프라인 탐지
│   ├── indexer.py            # ③ 검색 폴더 색인
│   ├── matcher.py            # ④ 매칭 + 신뢰도
│   ├── writer.py             # ⑤ 백업 + 경로 치환 + 저장
│   └── models.py             # 공용 데이터 구조 (MediaRef, Candidate, MatchResult 등)
├── skill/                    # 클로드 스킬 껍데기 (→ 추후 ~/.claude/skills로 배치)
│   └── SKILL.md              # 스킬 정의 + 오케스트레이션 설명
└── tests/                    # 엔진 단위 테스트 (실제 prproj 픽스처 기반)
```

## 4. 엔진 모듈 상세

### ① reader — 읽기/경로 추출

- 입력: `.prproj` 경로
- 처리: gzip 해제 → UTF-8 XML 문자열 확보 → 경로 태그(`ActualMediaFilePath`, `FilePath` 등)에서 **고유 경로 문자열 집합** 추출.
- 출력: `MediaRef` 목록 (경로 문자열, 등장 위치/횟수). 디코딩된 원본 XML 문자열도 함께 보관(치환용).
- 주의: 경로는 `/Users/...`, `/Volumes/<volume>/Users/...`, 드물게 `file://` 형태가 섞일 수 있음 → 모두 정규화하여 동일 물리 파일로 인식.

### ② detector — 오프라인 탐지

- 입력: `MediaRef` 목록
- 처리: 각 경로가 실제 디스크에 존재하는지 검사. 볼륨 접두(`/Volumes/Macintosh HD/...`)는 실경로로 환산 후 검사.
- 출력: **오프라인(누락) 경로만** 추린 목록. (온라인은 그대로 둠)

### ③ indexer — 검색 폴더 색인

- 입력: 검색 루트 목록 (기본: 프로젝트 폴더, `~/Desktop`, `~/Documents`, `~/Movies`)
- 처리: 루트들을 순회하며 `파일명(소문자) → [실제 경로...]` 색인 생성. 파일 크기도 함께 기록(크기 대조용).
- 출력: 검색 색인. (성능: 깊이/제외 폴더(`.Trash`, 캐시 등) 설정 가능)

### ④ matcher — 매칭 + 신뢰도

핵심 로직. 두 단계로 동작한다.

**4-1. 폴더 통째 이동 감지 (우선 적용) ⭐**
- 오프라인 경로들의 **공통 상위 경로 패턴**과, 색인에서 발견된 동일 파일들의 실제 위치를 비교해
  `옛 상위경로 → 새 상위경로` **치환 규칙**을 도출.
  - 예: `…/무제 폴더/0430 김보미 오부부/` → `…/0430 김보미 오부부/`
- 규칙 하나로 해당 폴더 하위 다수 파일을 일괄 재연결 → 빠르고 오연결 위험 낮음.
- 규칙 적용 후 실제 파일 존재까지 재확인하여 신뢰도 🟢로 분류.

**4-2. 개별 파일명 매칭 (잔여분)**
- 폴더 규칙으로 해결 안 된 오프라인 파일을 색인에서 파일명으로 탐색.

**신뢰도 판정**:

| 상황 | 신뢰도 | 동작 |
|------|--------|------|
| 폴더 이동 규칙 적용 + 실파일 확인 | 🟢 확실 | 자동 연결 |
| 파일명 일치 후보 **유일** (또는 크기까지 일치) | 🟢 확실 | 자동 연결 |
| 같은 이름 후보 **복수** / 크기 불일치 | 🟡 애매 | 사용자에게 후보 제시·확인 |
| 후보 없음 | 🔴 실패 | 미수정, "삭제 추정" 리포트 |

- 출력: `MatchResult` 목록 (각 오프라인 경로 → 신뢰도 + 후보(들)).

### ⑤ writer — 백업 + 치환 + 저장

- **백업 우선(무조건)**: 원본을 `프로젝트명.backup-YYYYMMDD-HHMMSS.prproj`로 복사.
- **수술적 치환**: XML 전체를 재직렬화하지 않고, 확정된 `옛 경로 문자열 → 새 경로 문자열` 쌍만 디코딩된 XML에서 정확히 치환. (의도한 바이트만 변경 → 프리미어가 못 여는 사고 방지)
  - 경로 문자열의 모든 형태(볼륨 접두 변형 포함)와 XML 이스케이프(예: `&amp;`)를 고려해 치환.
- **새 파일 저장(기본값)**: 치환된 XML을 다시 gzip 압축해 `프로젝트명_relinked.prproj`로 저장. 원본은 그대로 보존.
  - 옵션: 원본 위치 덮어쓰기 모드도 지원(백업이 있으므로 안전). 기본은 새 파일.
- 출력: 저장 경로 + 적용된 치환 요약.

## 5. 동작 흐름 (사용자 시점)

1. 사용자: "이 프로젝트(`…/0430 김보미 오부부.prproj`) 미디어 재연결해줘"
2. **백업** 생성
3. **읽기 → 오프라인 탐지**: "총 N개 미디어 중 M개가 오프라인입니다"
4. **색인 → 매칭**
5. **미리보기**:
   - 🟢 자동 연결 예정: K개 (요약 표시)
   - 🟡 확인 필요: L개 (후보 목록 제시 → 사용자 선택)
   - 🔴 못 찾음(삭제 추정): P개 (파일명 목록 표시, 미수정)
6. 사용자가 🟡 항목 확인/선택
7. **저장** → `프로젝트명_relinked.prproj`
8. **검증 리포트**: "재연결 완료 K+선택분 / 여전히 오프라인 P개. `_relinked` 파일을 프리미어에서 여세요."

## 6. 안전 / 에러 처리

- 원본 `.prproj`는 **절대 직접 덮어쓰지 않음**(기본). 백업 + 새 파일 저장의 이중 보호.
- gzip 해제 실패·비표준 포맷 → 중단하고 사용자에게 보고(원본 무손상).
- 치환 0건(고칠 게 없음) → 저장 생략, 그대로 보고.
- 검색 범위에서 못 찾은 파일 → 강제 연결 금지, 반드시 리포트로만.
- 대용량(42MB+ XML, 수천 경로) 대비: 색인·치환은 문자열/해시 기반으로 메모리 효율 고려.

## 7. 성공 기준 (검증 가능)

1. `0430 김보미 오부부` 프로젝트에 대해 실행 시, 폴더 이동(`무제 폴더` → 밖)으로 깨진 경로가 🟢 자동 재연결된다.
2. 생성된 `_relinked.prproj`를 프리미어에서 열면 해당 미디어가 온라인으로 표시된다.
3. 원본 `.prproj`와 타임스탬프 백업이 그대로 존재한다(무손상).
4. 못 찾은 파일은 수정되지 않고 목록으로 보고된다.
5. 엔진 모듈(①~⑤)이 클로드 비의존으로 단위 테스트 가능하다 (추후 CLI/GUI 재사용 가능).

## 8. 테스트 전략

- 실제 `.prproj` 샘플(자동저장본)을 픽스처로 사용하되, **복사본**으로만 테스트(원본 보호).
- 모듈별 단위 테스트:
  - reader: 알려진 파일에서 경로 N개 추출되는지
  - detector: 존재/누락 경로 분류 정확도
  - matcher: 폴더 이동 규칙 도출, 신뢰도 분류 (🟢/🟡/🔴)
  - writer: 백업 생성, 수술적 치환 정확성, gzip 재압축 후 프리미어가 여는지(수동 1회 확인)
- 통합 테스트: 일부러 경로를 깨뜨린 복사본 → 재연결 → 검증.
